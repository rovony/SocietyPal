📄 Step 16

Why test production build locally?

Testing production build process locally prevents deployment failures. The --no-dev --prefer-dist --optimize-autoloader flags exactly simulate what happens in production deployment.

Clean Previous Builds

bash

Purpose: Ensures we're testing the build process from a clean state, just like CI/deployment servers do.

Test Production PHP Build
Install production-only dependencies with optimizations
bash
echo "🔄 Testing production build process..." # Install production-only PHP dependencies (V1's approach) composer install --no-dev --prefer-dist --optimize-autoloader

Flag Explanations:

  • --no-dev: Excludes development dependencies (testing tools, debuggers)
  • --prefer-dist: Downloads stable releases, not development versions
  • --optimize-autoloader: Creates optimized class map for faster loading

  • Optimized autoloader generated for performance
  • No development tools included (PHPUnit, debuggers, etc.)

Restore Development Environment
Return to development mode after testing - Decision based on deployment method

📋 When to Run This Step

✅ RUN if: You plan to build again on a Build machine (GitHub Actions, DeployHQ, etc.)

❌ SKIP if: You plan to build locally and SFTP/zip to server - do that deployment first, then run this step later

⚠️ Important: If you skip this step and deploy locally-built files, you MUST run these commands after deployment if you plan to continue local development.

bash
# Clear production caches php artisan config:clear php artisan route:clear php artisan view:clear # Restore development dependencies (V1's approach) composer install npm install 2>/dev/null || true echo "✅ Development environment restored"

🔍 Deployment Method Guide

Local Build + Zip/SFTP: Skip this step → Deploy locally-built files first → Run commands if continuing development

Build Machine (GitHub Actions, DeployHQ): Run this step → Build machine will rebuild fresh from repo

🔧 Technical Context: Why This Matters

During production testing, your local environment changes:

  • ❌ Development tools removed (PHPUnit, debuggers)
  • ❌ Config cached (can't use .env changes)
  • ❌ Routes cached (route changes won't work)
  • ❌ Views cached (Blade changes won't show)

Environment Isolation: Your local state doesn't affect CI/CD runners (they start fresh), but DOES affect your ability to continue local development. Choose based on whether YOU will work locally after deployment.

Result: Production build tested and working, environment state matches your deployment workflow.

Step 18: Data Persistence System
Reference Guide

💾 Why This Step is Critical

Zero Data Loss: This step implements automated backup systems that ensure no data is ever lost during deployments, updates, or migrations. Data loss can be catastrophic for a business application.

Implement Automated Backup System
Create comprehensive backup scripts for database and user uploads
bash
echo "💾 Setting up automated backup system..." # Create backup script directory mkdir -p Admin-Local/0-Setup-Operations/backup-scripts cd Admin-Local/0-Setup-Operations/backup-scripts # Create comprehensive backup script cat > backup_all.sh << 'EOF' #!/bin/bash set -e # Configuration BACKUP_DIR="../backups_local/$(date +%Y-%m-%d_%H-%M-%S)" mkdir -p "$BACKUP_DIR" echo "🗄️ Starting comprehensive backup..." # Database backup if [ -f "../../../.env" ]; then source ../../../.env mysqldump -h"${DB_HOST:-localhost}" -u"$DB_USERNAME" -p"$DB_PASSWORD" "$DB_DATABASE" > "$BACKUP_DIR/database.sql" echo "✅ Database backed up" else echo "❌ .env file not found - cannot backup database" fi # User uploads backup if [ -d "../../../public/user-uploads" ]; then cp -r ../../../public/user-uploads "$BACKUP_DIR/" echo "✅ User uploads backed up" fi # Storage files backup if [ -d "../../../storage/app" ]; then cp -r ../../../storage/app "$BACKUP_DIR/storage_app" echo "✅ Storage files backed up" fi # Configuration backup cp ../../../.env "$BACKUP_DIR/env_backup" 2>/dev/null || echo "⚠️ No .env to backup" cp -r ../../../config "$BACKUP_DIR/config_backup" # Create backup manifest echo "# Backup Manifest" > "$BACKUP_DIR/BACKUP_INFO.md" echo "Date: $(date)" >> "$BACKUP_DIR/BACKUP_INFO.md" echo "Type: Full system backup" >> "$BACKUP_DIR/BACKUP_INFO.md" echo "Size: $(du -sh "$BACKUP_DIR" | cut -f1)" >> "$BACKUP_DIR/BACKUP_INFO.md" echo "✅ Complete backup created: $BACKUP_DIR" EOF chmod +x backup_all.sh echo "✅ Backup system created and made executable"

Expected Result:

  • Automated backup_all.sh script for database, uploads, and config
  • Timestamped backup directories for version tracking
  • Backup manifest with metadata for easy restoration
Test Backup and Restore Process
Verify backup integrity and restore functionality
bash
echo "🧪 Testing backup and restore process..." # Test backup creation cd Admin-Local/0-Setup-Operations/backup-scripts ./backup_all.sh # Verify backup integrity LATEST_BACKUP=$(ls -1 ../backups_local/ | tail -1) BACKUP_PATH="../backups_local/$LATEST_BACKUP" echo "🔍 Verifying backup integrity..." # Check database backup if [ -f "$BACKUP_PATH/database.sql" ]; then echo "✅ Database backup exists ($(wc -l < "$BACKUP_PATH/database.sql") lines)" else echo "❌ Database backup missing" fi # Check user uploads backup if [ -d "$BACKUP_PATH/user-uploads" ]; then echo "✅ User uploads backup exists ($(find "$BACKUP_PATH/user-uploads" -type f | wc -l) files)" else echo "ℹ️ No user uploads to backup" fi # Create restore script cat > restore_backup.sh << 'EOF' #!/bin/bash set -e if [ -z "$1" ]; then echo "Usage: ./restore_backup.sh " echo "Available backups:" ls -1 ../backups_local/ exit 1 fi BACKUP_PATH="../backups_local/$1" if [ ! -d "$BACKUP_PATH" ]; then echo "❌ Backup directory not found: $BACKUP_PATH" exit 1 fi echo "🔄 Restoring from backup: $1" # Restore database if [ -f "$BACKUP_PATH/database.sql" ]; then source ../../../.env mysql -h"${DB_HOST:-localhost}" -u"$DB_USERNAME" -p"$DB_PASSWORD" "$DB_DATABASE" < "$BACKUP_PATH/database.sql" echo "✅ Database restored" fi # Restore user uploads if [ -d "$BACKUP_PATH/user-uploads" ]; then rm -rf ../../../public/user-uploads cp -r "$BACKUP_PATH/user-uploads" ../../../public/ echo "✅ User uploads restored" fi echo "✅ Restore completed successfully" EOF chmod +x restore_backup.sh echo "✅ Backup and restore system tested and ready"

🔍 Educational Context

Disaster Recovery: A backup is only as good as your ability to restore from it. Testing the restore process ensures you can actually recover your data when needed.

Create Persistence Verification System
Automated verification that data persists through deployment cycles
bash
echo "🔍 Creating persistence verification system..." # Create verification script cat > verify_persistence.sh << 'EOF' #!/bin/bash set -e echo "🔍 Verifying data persistence..." # Check database connectivity and basic data if [ -f "../../../.env" ]; then source ../../../.env # Test database connection mysql -h"${DB_HOST:-localhost}" -u"$DB_USERNAME" -p"$DB_PASSWORD" -e "USE $DB_DATABASE; SHOW TABLES;" > /dev/null echo "✅ Database connection verified" # Check for critical tables TABLES=$(mysql -h"${DB_HOST:-localhost}" -u"$DB_USERNAME" -p"$DB_PASSWORD" -e "USE $DB_DATABASE; SHOW TABLES;" | wc -l) echo "✅ Database has $TABLES tables" # Check for user data USERS=$(mysql -h"${DB_HOST:-localhost}" -u"$DB_USERNAME" -p"$DB_PASSWORD" -e "USE $DB_DATABASE; SELECT COUNT(*) FROM users;" | tail -1) echo "✅ Database has $USERS users" else echo "❌ Cannot verify database - .env file missing" fi # Check file system persistence echo "📁 Checking file system persistence..." # Check storage directories exist and are writable for dir in "storage/logs" "storage/app" "storage/framework"; do if [ -d "../../../$dir" ] && [ -w "../../../$dir" ]; then echo "✅ $dir is writable" else echo "❌ $dir is not writable" fi done # Check user uploads directory if [ -d "../../../public/user-uploads" ]; then UPLOADS=$(find ../../../public/user-uploads -type f | wc -l) echo "✅ User uploads directory has $UPLOADS files" else echo "ℹ️ No user uploads directory (will be created on first upload)" fi echo "✅ Persistence verification completed" EOF chmod +x verify_persistence.sh # Test the verification script ./verify_persistence.sh echo "✅ Persistence verification system created and tested"

Expected Result: Automated system to verify data integrity and persistence across deployments.
